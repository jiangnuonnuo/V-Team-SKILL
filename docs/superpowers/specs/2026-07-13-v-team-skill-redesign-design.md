# V-Team 协作开发技能重构设计

## 1. 设计目标

本次重构把现有 `project-harness-lite` 从“按版本目录组织的项目管理工具”改造成面向个人或小团队自用的多 Agent 协作开发技能。一个代码仓库代表一个完整项目，不再创建 `Plan/versions/v1`、`Plan/versions/v2` 等版本目录。

重构后的技能必须满足以下目标：

1. 同一项目中允许多个 Codex 或 Claude Agent 承担不同角色、负责不同模块。
2. 每个 Agent 都拥有明确的职责、代码范围和路径白名单，同时允许用户对一次跨范围提交进行临时授权。
3. Codex 使用项目根目录 `AGENTS.md`，Claude 使用项目根目录 `CLAUDE.md`；两者都必须要求 Agent 读取自己的 `Plan/agents/<agent-id>/AGENT.md`。
4. 每个 Agent 只维护一份活动计划 `PLAN.md`。计划审批、任务项、测试结果、本地提交和整体完成状态都保存在同一文件中。
5. 任务项粒度是一个完整功能或一个明确功能修复，不能拆成机械、零散、没有独立用户价值的代码动作。
6. 用户批准计划前禁止进入代码实现。每个任务项实现后必须进行必要测试，测试通过后才能使用中文提交到本地 Git。
7. 技能禁止 Agent 自行推送远程或合并代码，但不创建、不切换、不命名也不管理 Git 分支。
8. 不同 Agent 通过少量固定的当前态协作文档共享架构、接口契约和对接状态，不依赖彼此的聊天上下文。
9. 已完成、已废弃和过期内容退出活动上下文，并采用归档优先的安全清理策略。
10. 所有脚本只依赖 Python 标准库和 Git，同时支持 Windows 与 macOS。

## 2. 非目标

本次重构不实现以下能力：

- 不管理 Git 分支，不为 Agent 创建独立分支，也不规定分支命名。
- 不自动推送远程、不自动合并、不创建 Pull Request。
- 不在每次文件修改前执行路径检查。
- 不安装 Git Hook，不修改用户的全局或项目 Git 配置。
- 不保存完整聊天记录，不为每轮对话创建新文档。
- 不恢复现有的版本目录、经理周期报告、版本归档或 solo 角色轮换结构。
- 不把简单跨模块修改强制转换成正式对接流程。

## 3. 项目目录

初始化后的项目使用固定、无版本的目录结构：

```text
<project-root>/
  AGENTS.md                         # 项目存在 Codex Agent 时生成
  CLAUDE.md                         # 项目存在 Claude Agent 时生成
  Plan/
    project.md                      # 项目目标、模块边界、当前阶段和全局约束
    team.json                       # Agent 身份、运行端、职责和写入白名单
    agents/
      <agent-id>/
        AGENT.md                    # 个人身份、必读规则、边界和协作要求
        PLAN.md                     # 该 Agent 唯一的活动计划文档
    collaboration/
      architecture.md               # 当前有效架构，由架构职责 Agent 主要维护
      api-contracts.md              # 当前有效接口契约，由接口生产方主要维护
      handoffs.md                   # 仍未完成的跨 Agent 对接事项
    archive/                        # 已完成或已废弃内容，默认不进入 Agent 上下文
```

`Plan/project.md`、`architecture.md`、`api-contracts.md` 和 `handoffs.md` 都是当前态文档。新需求优先覆盖或修订现有内容；只有需要保留稳定决策、完成证据或废弃原因时才归档。

## 4. Agent 身份与运行端适配

### 4.1 根目录约束文件

技能根据 `team.json` 中已注册 Agent 的运行端生成入口文件：

- 存在 `runtime: codex` 的 Agent 时生成 `AGENTS.md`。
- 存在 `runtime: claude` 的 Agent 时生成 `CLAUDE.md`。
- Codex 与 Claude 混合协作时同时生成两个文件。

两个入口文件使用各自产品识别的文件名，但表达相同的项目行为约束：

1. Agent 必须从用户指令或当前任务上下文确定自己的 `agent-id`。
2. 如果没有明确 `agent-id`，必须停止开发并询问用户，不能自行冒充其他身份。
3. 确定身份后必须读取 `Plan/agents/<agent-id>/AGENT.md`。
4. 只读取自己的活动 `PLAN.md`、`Plan/project.md` 以及个人规则指定的协作文档。
5. 默认忽略 `Plan/archive/`、状态为 `completed` 或 `abandoned` 的计划，以及与自身无关的历史内容。
6. 用户未批准活动计划前不得实现代码。
7. 每个任务项测试通过后才能进入本地提交流程。
8. 禁止自行推送远程或合并代码。

### 4.2 个人 `AGENT.md`

每个 Agent 的个人文件至少包含：

- Agent ID、运行端、角色和职责描述。
- 主要负责模块和允许提交的路径白名单。
- 明确禁止或需要协调的范围。
- 必须读取的当前态文档。
- 允许直接完成跨模块修改的条件。
- 白名单外提交的一次性用户授权流程。
- 计划审批、测试、中文本地提交和状态更新规则。
- 协作文档的生产与消费责任。
- 中断恢复时的最小读取顺序。
- 默认忽略已完成、已废弃和归档文件的规则。

个人规则是行为入口，`team.json` 是路径校验的数据来源。两者内容必须保持一致；重新生成约束文件时，以 `team.json` 为身份和白名单事实源。

## 5. 白名单与跨模块修改

### 5.1 默认边界

每个 Agent 优先在自己的职责模块内工作。白名单限制的是“未经用户确认可以直接提交的路径”，不是完全禁止读取其他模块，也不是禁止 Agent 理解或修改跨模块功能。

### 5.2 提交前检查

Agent 完成一个任务项并通过测试后，按以下顺序处理：

1. 暂存本任务项需要提交的文件。
2. 使用 `git diff --name-only HEAD` 获取当前提交涉及的全部路径。
3. 将 Git 返回的 `/` 分隔相对路径与 `team.json` 中当前 Agent 的白名单匹配。
4. 全部路径在白名单内时，允许继续中文本地提交。
5. 存在白名单外路径时暂停提交，向用户列出越权路径、修改原因、影响模块和建议提交说明。
6. 用户批准后，在当前 `PLAN.md` 记录一次性授权，再允许当前提交。
7. 一次性授权只对当前提交有效，不自动修改永久白名单。
8. 用户拒绝时不得提交相关路径；Agent 应撤销、拆出或交由更合适的 Agent 处理。

技能不在每次写文件前执行检查，也不安装 Git Hook。白名单检查只发生在本地提交前。

## 6. 单份计划与用户审批

每个 Agent 固定使用 `Plan/agents/<agent-id>/PLAN.md`，不为每个对话、新需求或修复创建新的活动文件。

### 6.1 计划状态

计划使用以下状态：

- `draft`：正在分析和拆分。
- `waiting-approval`：等待用户审查。
- `in-progress`：用户已批准，正在实现。
- `completed`：所有任务项、测试、提交和必要协作均已完成。
- `abandoned`：用户明确放弃或方案已失效。

### 6.2 计划内容

同一份计划必须包含：

1. 当前目标和用户需求摘要。
2. 范围、非目标、影响模块和预计修改路径。
3. 验收标准和风险。
4. Review 结论、用户批准状态和批准记录。
5. 任务项表格。
6. 每个任务项的测试方法、测试结果、状态和本地提交哈希。
7. 当前阻塞、下一步动作和协作依赖。
8. 白名单外路径的一次性授权记录。
9. 整体完成结论。

### 6.3 任务项粒度

一个任务项必须是以下两类之一：

- 可以独立验收的完整功能。
- 可以独立验证的明确功能修复。

不能把“创建文件”“新增一个方法”“修改配置”“补一行注释”等机械动作单独作为任务项，除非它本身就是用户要求的完整交付结果。一个任务项完成后应适合形成一次语义完整的本地 Git 提交。

### 6.4 执行顺序

1. Agent 分析需求并直接更新自己的 `PLAN.md`。
2. Agent 将计划状态设为 `waiting-approval` 并等待用户审查。
3. 用户批准后将计划设为 `in-progress`，然后才能实现代码。
4. 每次只执行一个未完成任务项。
5. 完成实现后运行足以证明功能正确的测试，不要求无关的复杂测试体系。
6. 测试失败时不得提交，任务项保持未完成并记录失败原因。
7. 测试通过后执行提交前白名单检查。
8. 完成中文本地提交后，将任务项标记为完成并记录测试结果和提交哈希。
9. 所有任务项完成后，处理必要协作文档，将整个计划设为 `completed`。
10. 执行安全清理，使已完成内容退出活动上下文。

## 7. 多 Agent 协作

协作采用“默认归属、允许跨界、必要时契约化”的原则，不把边界约定得过死。

### 7.1 默认行为

- Agent 优先修改自己负责的模块。
- 如果跨模块修改更直接，而且 Agent 能确认调用关系、影响范围和测试方式，可以直接完成。
- 跨模块路径在提交前按照一次性授权规则处理。
- 简单、一次性的跨模块修改不强制创建对接事项。

### 7.2 需要协作文档的情况

只有出现以下情况时才更新协作文档：

- 一个 Agent 的产物将长期被另一个 Agent 消费。
- 接口、数据结构、模块边界或运行方式发生变化。
- 另一个 Agent 必须完成后续集成才能形成完整功能。
- 当前存在尚未解决的跨模块依赖或验收责任。

### 7.3 固定协作文档

- `architecture.md` 保存当前有效模块边界、调用关系、数据流和技术约束。架构职责 Agent 是主要维护者，其他 Agent 可以在理解影响后修订，并在白名单外提交时申请授权。
- `api-contracts.md` 保存当前有效接口、字段、错误约定和调用示例。接口生产方负责保持内容与实现一致，消费方负责记录集成结果或问题。
- `handoffs.md` 只保存未完成的跨 Agent 对接事项，包含提出者、接收者、交付物、验收条件和状态。

对接完成并验证后，活动事项从 `handoffs.md` 移除，精简摘要进入归档。新 Agent 不读取已经关闭的对接历史。

### 7.4 新 Agent 接入顺序

新 Agent 只按以下顺序加载上下文：

1. 根目录 `AGENTS.md` 或 `CLAUDE.md`。
2. 自己的 `AGENT.md`。
3. 自己的活动 `PLAN.md`。
4. `Plan/project.md`。
5. 个人 `AGENT.md` 指定、并且与当前任务直接相关的协作文档。

这保证 Agent 能快速恢复工作，同时避免把全项目历史和其他 Agent 的已完成计划装入上下文。

## 8. 当前态文档与清理

### 8.1 覆盖优先

- `project.md`、个人 `PLAN.md` 和协作文档保存当前事实。
- 需求澄清、方案修正、接口变化和进度变化优先修改原有段落。
- 只有稳定决策、完成证据、废弃原因和必要追溯信息才追加到归档。
- 不生成 `final-2`、`review-new`、`task-copy` 等重复文件。

### 8.2 清理条件

统一清理入口只在以下条件满足时执行：

- `PLAN.md` 状态为 `completed` 或 `abandoned`。
- 已完成任务项包含测试结果；正常完成的任务项包含本地提交哈希。
- 需要其他 Agent 消费的契约已经更新。
- 相关对接事项已经关闭或明确转移。

### 8.3 清理行为

清理采用归档优先：

1. 把完成计划的精简快照写入 `Plan/archive/`。
2. 将个人活动 `PLAN.md` 重置为无活动任务状态，供下一个需求复用。
3. 从 `handoffs.md` 移除已验证或已取消事项。
4. 从当前态文档删除已废弃思路、过期候选方案和重复描述，只保留最终有效结论。
5. 不永久删除仍有追溯价值的完成证据。
6. 根约束文件明确禁止 Agent 默认读取归档。

## 9. 技能实现结构

现有 5 个围绕版本目录的脚本将被替换为一个统一入口 `scripts/vteam.py`，避免 Agent 在日常工作中反复调用多套脚本。

统一入口只保留四类能力：

- `init`：初始化无版本 `Plan/`，根据已配置运行端生成 `AGENTS.md` 或 `CLAUDE.md`。
- `agent`：新增或更新 Agent 身份、白名单、个人 `AGENT.md` 和活动 `PLAN.md`。
- `check-scope`：本地提交前检查当前 Agent 的变更路径，输出白名单外文件。
- `cleanup`：在计划完成或废弃后归档精简证据、重置活动计划并清理关闭的对接事项。

计划审批、任务项更新、测试记录和协作内容由 Agent 按模板直接维护在 Markdown 文件中，不额外创建状态更新脚本。

技能资源调整为：

```text
SKILL.md
agents/openai.yaml
scripts/vteam.py
references/
  root-agents-template.md
  root-claude-template.md
  personal-agent-template.md
  plan-template.md
  project-template.md
  architecture-template.md
  api-contracts-template.md
  handoffs-template.md
  team-template.json
tests/
  test_vteam.py
README.md
```

旧的版本化模板和脚本在新实现验证通过后删除，不保留兼容层，不生成迁移版本目录。

## 10. 异常处理

- 无法确定 Agent ID：停止实现并询问用户。
- `team.json` 缺失或格式错误：禁止生成约束或提交，指出具体字段问题。
- 个人 `AGENT.md` 缺失：禁止进入计划和开发，要求先注册或重新生成。
- 计划未获用户批准：禁止实现代码。
- 测试失败：禁止提交，保留当前任务项状态并记录失败原因。
- 白名单外路径存在：暂停提交并请求一次性用户授权。
- 用户拒绝越权提交：不得提交相关路径，需要撤销、拆分或转交。
- 本地提交失败：任务项不能标记完成，也不能填写虚假提交哈希。
- 清理活动计划：直接拒绝，只有 `completed` 或 `abandoned` 才允许清理。
- 协作契约不完整：不得推断为已经完成，保持对接事项未关闭。

## 11. 跨平台要求

- 使用 Python 3 标准库，不引入 PyYAML、第三方 CLI 或系统专用依赖。
- 所有路径通过 `pathlib` 处理；配置文件中的白名单统一使用 Git 风格 `/` 分隔路径。
- Git 命令通过 `subprocess.run([...])` 参数列表执行，不拼接 shell 命令。
- 文件统一使用 UTF-8 和 `\n`，中文提交信息按 UTF-8 传递。
- 不依赖软链接、Bash、PowerShell、AppleScript 或注册表。
- Windows 盘符、反斜杠和大小写差异在进入白名单匹配前统一规范化。
- macOS 保持大小写敏感的真实路径语义，但配置比较不允许路径逃逸项目根目录。

### 11.1 实现代码规范

- 保持线性、易读的实现方式，不使用不必要的复杂嵌套、动态技巧或过度抽象。
- 每个 Python 模块、类和方法都必须提供准确的中文 `description` 或 docstring。
- 方法 docstring 必须说明输入参数、返回结果和可能抛出的异常。
- 关键执行步骤使用独立行注释解释目的，不使用行尾注释。
- 逻辑确有独立职责时才拆分函数，不为追求形式增加无意义调用层级。

## 12. 测试设计

实现阶段使用 Python `unittest` 和临时目录完成跨平台测试，不依赖真实用户项目。

必须覆盖：

1. Codex 项目只生成 `AGENTS.md`。
2. Claude 项目只生成 `CLAUDE.md`。
3. 混合团队同时生成两个入口文件。
4. 入口文件包含“必须读取个人 `AGENT.md`”“计划批准前禁止开发”“禁止自行推送或合并”“忽略完成和归档内容”等核心规则。
5. 注册多个同角色 Agent 时分别生成独立身份、职责和白名单。
6. 不创建任何 `Plan/versions/` 目录。
7. 白名单内路径通过检查。
8. 白名单外路径被准确列出并要求用户授权。
9. 一次性授权不写回永久白名单。
10. Windows 反斜杠、Git `/` 路径、空格和非 ASCII 文件名可以正确处理。
11. 活动计划不能清理。
12. 完成或废弃计划能够归档、重置，并从活动对接文档移除关闭事项。
13. 未测试、未提交或协作未闭环的计划不能被错误标记为正常完成。
14. 旧版本目录和旧脚本不会被新初始化流程重新生成。

## 13. 实施边界

本轮实现只修改 `V-Team-SKILL` 本地 `develop` 分支并进行本地中文提交，不推送远程、不合并分支。源码验证完成后，是否同步到用户已安装的 Codex 技能目录由用户另行确认。

技能安装标识继续使用现有 `project-harness-lite`，避免在本次行为重构中同时引入无关的技能重命名。

技能库索引和说明文件必须随最终技能更新同步维护：更新仓库内 `README.md`，并同步更新 `F:\java\code\SKILLS\README.md` 中的 `project-harness-lite` 条目，保证技能名称、用途、目录和实际实现一致。
