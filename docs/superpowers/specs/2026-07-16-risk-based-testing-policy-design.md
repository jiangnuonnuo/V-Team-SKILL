# V-Team 风险分级测试与验收 Review 设计

## 目标

消除 V-Team 中“每个细小改动都重新测试”的倾向，同时保留与风险相称的验证证据。Review 聚焦需求、功能结果与已有证据，不把重复执行测试作为默认职责。

## 非目标

- 不取消测试、提交前范围检查或高风险变更的完整回归。
- 不修改 `vteam.py` 的审批、范围检查和归档流程。
- 不把外部接口的真实调用变成无条件重试。

## 测试策略

| 等级 | 适用变更 | 所需验证 |
|---|---|---|
| 定向验证 | 单模块、低风险功能或明确修复 | 受影响功能的最小自动化或手动验证；必要时加静态检查。 |
| 相关完整回归 | 跨模块、公共接口、安全、数据迁移 | 受影响模块、直接依赖模块、接口契约/集成测试及必要构建、类型或静态检查。 |
| 全仓回归 | 发布、全局基础设施或全局配置 | 全仓自动化测试；必要时端到端测试。 |

外部真实接口验证必须限频；失败后先记录证据并分析原因，不进行无条件自动重试。

## 证据复用

同一代码、依赖和运行配置未变化时，一次成功的定向测试或 TDD Green 结果可同时作为任务完成和提交前验证证据。只有受测代码、相关依赖、配置或测试环境发生变化时，才重新运行对应验证。

## Review 规则

Review 必须检查：需求与非目标、验收标准、改动范围与风险、测试等级和命令、结果证据以及失败/例外处理。

Reviewer 默认审阅当前 diff 和已有测试结果，不重复执行测试。仅在以下情况补充验证：证据缺失或过期、结果与验收标准冲突、变更属于相关完整回归或全仓回归等级、或发现新的高风险影响。

## 模板记录

`PLAN.md` 的 Review 记录新增或明确以下字段：测试等级、测试命令与结果、证据有效范围。功能任务表继续记录最终验证结果，不把相同代码状态下的重复命令视为独立必填项。
